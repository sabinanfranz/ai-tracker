# Claude Code 카드 한국어 번역 스타일 가이드

## title_kor 규칙

### 어미
- **~해요** / **~됐어요** / **~있어요** / **~나왔어요** / **~돼요** 체 사용
- 절대 금지: ~습니다, ~하였음, ~됨 (딱딱한 어미)

### change_type별 어미 패턴

| change_type | 어미 패턴 | 예시 |
|-------------|-----------|------|
| added | ~추가됐어요 | "MCP 서버 지원이 추가됐어요" |
| changed | ~변경됐어요 / ~개선됐어요 | "터미널 출력이 개선됐어요" |
| removed | ~제거됐어요 | "레거시 API가 제거됐어요" |
| deprecated | ~곧 사라져요 | "이전 설정 방식이 곧 사라져요" |
| other | ~됐어요 / ~돼요 | "새 버전이 출시됐어요" |

### 길이
- 목표: 30자 내외 (최소 10자, 최대 40자)

### 구조
- 사용자 관점에서 **무엇이 달라졌는지** 한 문장으로 전달
- 기능 나열이 아닌, 사용자 경험 중심 서술

### 금지 패턴
- 키워드 나열: "A, B 및 C" / "A, B, C 지원"
- 명사 종결: "~출시" / "~추가" / "~제거"
- 개발 용어 직역: "비동기 압축 개선" → "대화가 더 빠르게 정리돼요"

### 고유명사
- 도구명: Claude Code, MCP, VSCode, JetBrains 등 영어 유지
- 프로토콜/기술: API, SDK, CLI, IDE, JSON, YAML 등 영어 유지
- 서비스명: GitHub, Slack, Linear 등 영어 유지

## content_kor 규칙

### 형식
- 정확히 **불릿 3개**, `•` 접두사, `\n` 구분
- 각 불릿 1~2문장

### 어투
- **~해요** / **~돼요** / **~있어요** 체 (title_kor와 동일)

### 기술 용어 해설
- 첫 등장 시 괄호로 해설: "MCP(Model Context Protocol)", "LSP(Language Server Protocol)"
- 2회차부터는 약어만 사용

### 불릿 구조
- 1번째: 핵심 변경 사항 (이 기능이 뭔지)
- 2번째: 부가 설명 또는 세부 동작
- 3번째: 적용 범위, 호환성, 또는 참고 사항

### 길이
- 각 불릿 30~80자

## 변환 사고 과정

1. 원문(title) 읽기
2. change_type 확인 → 어미 결정
3. 핵심 변경 1개 추출
4. 사용자에게 어떤 영향인지 재구성
5. ~해요/~됐어요 어미로 문장 완성
6. 30자 내외로 다듬기

## content_kor 작성 사고 과정

1. title + subsystem 정보 확인
2. 변경의 what/how/where 분리
3. 각 항목을 • 불릿으로 작성
4. 기술 용어 첫 등장 시 괄호 해설 추가
5. 각 불릿 30~80자 검증
